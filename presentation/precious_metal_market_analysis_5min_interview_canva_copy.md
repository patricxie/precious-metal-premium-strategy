# 貴金屬市場分析 ETL 專案｜面試版 Canva 文案

## 使用方向
- 目標：面試展示，不是課堂型長篇報告
- 風格：短句、強調工程決策、強調可落地成果
- 建議頁數：8 頁
- 建議節奏：每頁 25 到 40 秒

## 第 1 頁｜封面
標題：
貴金屬市場分析 ETL 專案

副標：
從多來源市場資料到可查詢交易訊號的資料流程實作

補充小字：
Gold / Silver Futures + Spot Data
ETL Pipeline / Premium / Z-score / SQLite

一句話定位：
這個專案的重點不是只做分析，而是把分析題目做成可重複執行的資料產品。

## 第 2 頁｜我想解決的問題
頁面標題：
Why This Project

頁面文案：
- 市場資料常來自不同來源，格式與日期不一致，直接分析容易失真
- 我想驗證自己能不能把一個分析題目，真正落地成 ETL pipeline
- 專案核心問題：期貨相對現貨的偏離，能否轉成可量化的交易訊號

頁尾一句：
重點不是預測市場，而是建立一套可重複、可驗證、可擴充的資料流程

## 第 3 頁｜我實作了什麼
頁面標題：
What I Built

頁面文案：
- Extract：抓取黃金與白銀期貨資料，以及現貨 API 資料
- Transform：標準化欄位、對齊日期、計算 Premium 與 Z-score
- Load：輸出 CSV 並寫入 SQLite，供後續分析與展示使用
- Output：可接回測、圖表、報表與 dashboard

技術標籤：
Python
Pandas
SQLite
Matplotlib
Streamlit

## 第 4 頁｜資料與分析邏輯
頁面標題：
Data + Signal Logic

頁面文案：
- 資料來源：Yahoo Finance 期貨資料 + 現貨 API 資料
- Premium (%) = (Futures - Spot) / Spot × 100
- 使用 rolling Z-score 判斷 Premium 是否落在異常高低區間
- Z-score < -2：BUY
- Z-score > 2：SELL
- 其餘：HOLD

頁尾一句：
我把主觀判斷轉成可解釋、可重複的統計規則

## 第 5 頁｜工程上的挑戰與處理方式
頁面標題：
Engineering Challenges

左欄：
遇到的問題
- 不同資料源欄位名稱不一致
- 期貨與現貨日期需要對齊
- API 可能有缺值或格式異常
- 如果流程過度耦合，後續很難維護

右欄：
我的做法
- 先做 schema normalization，再進行合併
- 用一致的日期格式與 join 邏輯避免錯配
- 在 transform 階段處理缺值與 rolling 計算
- 將流程拆成 extract / transform / load 模組

頁尾一句：
這一頁要傳達的是，我有處理真實資料問題，而不只是跑出結果

## 第 6 頁｜專案成果
頁面標題：
Results

頁面文案：

- 結果可輸出成 CSV，也可存進 SQLite
- 可延伸到回測、圖表報告與前端展示

頁尾一句：
重點不是單一數字，而是資料流程已經可以持續產出分析結果

Why It Matters For Data Engineering
